Quick Connect

Connessioni istantanee senza salvare un profilo:

  1. Scegli il protocollo (SSH, Telnet, SFTP, FTP, RDP, VNC)
  2. Digita la destinazione: utente@host:porta oppure solo host
  3. Premi Invio o clicca ▶ Connetti
💡 La Quick Connect chiede le credenziali al volo se non specificate. Il formato utente@host:2222 imposta utente e porta in un colpo solo.

Sidebar sessioni

  • Sezione ⏱ Recenti in cima (senza filtro) — ultime 20 sessioni usate con data/ora dell'ultima connessione
  • Sessioni raggruppate per gruppo, ordinate alfabeticamente
  • Ogni riga mostra icona protocollo, nome e utente@host
  • Campo ricerca live — filtra per nome, host e utente
  • Tasto destro sul gruppo ⏱ Recenti → Cancella cronologia

Area schede (tab)

Ogni connessione apre una scheda. Puoi avere quante schede vuoi in parallelo.

  • Riordina le schede trascinandole con il mouse
  • Tasto destro sul tab — menu contestuale con: sposta nell'altro pannello, esporta apri-connessione.sh, esporta comandi.sh, chiudi
  • In modalità Split, le schede si distribuiscono tra i due pannelli

System Tray

Con sessioni attive, chiudere la finestra minimizza PCM nel tray invece di terminare le connessioni.

  • Doppio clic sull'icona — riapre la finestra principale
  • Tasto destro sull'icona — menu con accesso rapido alle ultime sessioni usate e opzione Esci
  • Il menu del tray mostra prima le ultime 8 sessioni usate, poi le restanti in ordine alfabetico

📋 Gestione Sessioni

Creare una nuova sessione

1
Clicca ➕ nella sidebar, oppure Ctrl+Shift+N.
2
Assegna un Nome descrittivo (es. Server produzione) e un Gruppo opzionale (es. Lavoro).
3
Scegli il Protocollo — il dialog si adatta mostrando solo i campi rilevanti.
4
Tab Connessione: Host, Porta, Utente, e le opzioni specifiche del protocollo.
5
Tab Autenticazione: password o chiave privata SSH. Puoi lasciare vuoto — PCM chiederà al momento della connessione.
6
Tab Terminale: tema colori, font, dimensione, comando avvio remoto, comando pre-connessione locale, log output.
7
Tab ⚙ Avanzate: keepalive, X11 forwarding, compressione, jump host, modalità di apertura, selettore terminale esterno.
8
Tab ⚡ Macro: definisci comandi rapidi inviabili con un clic dal pannello sessioni.
9
Tab 📊 Pannelli (SSH, VNC Linux, RDP Windows): abilita il pannello informazioni laterale destro e il browser SFTP laterale. Per SSH e VNC Linux sono disponibili CPU/RAM, processi, disco, rete, log streaming. Per RDP Windows il pannello usa PowerShell via OpenSSH (tab Log nascosto). Ogni sezione è attivabile individualmente.
10
Tab 📝 Note: campo testo libero per annotazioni, procedure, link utili.
11
Clicca OK — la sessione viene salvata e la connessione si apre immediatamente.

Menu contestuale sessione (tasto destro)

  • ▶ Connetti — apre la connessione in una nuova scheda
  • ✏ Modifica — riapre il dialog con i dati precompilati
  • 📋 Duplica — crea una copia con suffisso "(copia)"
  • 📋 Copia comando — copia il comando shell equivalente negli appunti
  • 📄 Esporta apri-connessione.sh… — genera uno script bash standalone per aprire la connessione
  • 📂 Apri SFTP/FTP qui — apre un browser file transfer relativo alla sessione SSH/Telnet/Mosh selezionata
  • Visualizza log… — apre un tab di streaming log (journalctl/tail -f) per la sessione SSH selezionata
  • Monitor sistema… — apre un tab di monitoraggio risorse (CPU, RAM, disco, rete) per la sessione SSH selezionata
  • Ping — verifica la raggiungibilità TCP dell'host sulla porta configurata (disponibile per sessioni con host di rete)
  • ⚡ Macro — sottomenu con le macro definite per questa sessione
  • 🗑 Elimina — richiede conferma

Importare ed esportare sessioni

Da File → Esporta sessioni salvi tutte le sessioni in un file JSON portabile. Da File → Importa sessioni carichi un file JSON precedentemente esportato. Le sessioni sono salvate in connections.json nella cartella di PCM — modificabile con qualsiasi editor.

🔌 Protocolli supportati

SSH Secure Shell

Connessione cifrata a terminale remoto. Il protocollo standard per l'amministrazione di server Linux/Unix.

  • Autenticazione — password oppure chiave privata (consigliata)
  • Jump host — server intermedio (bastion) per host non raggiungibili direttamente
  • X11 Forwarding — esegui applicazioni grafiche remote sul display locale
  • Keepalive — pacchetti periodici per mantenere viva la connessione su link instabili
  • Compressione — riduce il traffico, utile su connessioni lente
  • Comando avvio — eseguito subito dopo la connessione (es. tmux attach)
  • Pre-connessione locale — comando eseguito sul tuo PC prima di connettersi (es. attiva VPN)
  • Compatibilità legacy — opzioni per server datati (KEX, cipher, host key, MAC obsoleti)
  • Agent Forwarding (-A) — propaga le chiavi dell'ssh-agent locale ai server remoti (utile per hop multipli)
  • Modalità apertura — terminale interno embedded oppure terminale esterno
💡 Con l'opzione "Apri browser SFTP laterale" attiva, il pannello SFTP si popola automaticamente dopo la connessione.

SFTP SSH File Transfer Protocol

Trasferimento file sicuro su canale SSH. Non è FTP — usa una connessione SSH cifrata.

  • Stessi parametri di SSH (host, porta, utente, chiave/password)
  • Browser grafico integrato stile WinSCP: navigazione, upload, download, rinomina, eliminazione
  • Visualizzazione permessi, dimensioni, date di modifica
  • Modalità apertura — browser interno, terminale SFTP CLI, file manager di sistema

FTP / FTPS File Transfer Protocol

Trasferimento file non cifrato (FTP) o con TLS esplicito (FTPS). Preferisci SFTP quando disponibile.

  • FTPS — TLS esplicito (AUTH TLS): cifra credenziali e dati
  • Modalità passiva (PASV) — consigliata dietro NAT o firewall
  • Browser grafico integrato con upload, download, navigazione
  • Modalità apertura — browser interno, terminale lftp, file manager di sistema
⚠️ FTP plain trasmette credenziali e dati in chiaro. Usa FTPS o SFTP quando possibile.

RDP Remote Desktop Protocol

Desktop remoto Windows — mostra l'intero desktop della macchina remota.

  • Client RDP — xfreerdp3 (raccomandato), xfreerdp, rdesktop
  • Dominio / Workgroup — inserisci il dominio Active Directory o il workgroup (es. AZIENDA, WORKGROUP); lascia vuoto per macchine standalone
  • Modalità apertura — Finestra esterna (default, xfreerdp in finestra separata) oppure Pannello interno (desktop RDP incorporato nella scheda PCM, richiede X11 o XWayland)
  • Schermo intero — Ctrl+Alt+Invio per uscire (solo finestra esterna)
  • Monitor — singolo (default), tutti i monitor (/multimon), o selezione personalizzata per ID (/monitors:0,1)
  • Condivisione clipboard e cartelle locali
ℹ️ Il combo Client RDP mostra solo i client installati. rdesktop è il client raccomandato per il pannello interno (usa -X per embedding nativo diretto, connessione rapida senza Kerberos). xfreerdp3 supporta il pannello interno tramite xdotool ma richiede più tempo. Con xfreerdp v2 il pannello interno è disabilitato. Il campo Autenticazione permette di scegliere NTLM (veloce, senza timeout Kerberos) o Kerberos+NTLM (standard AD).

VNC Virtual Network Computing

Desktop remoto multipiattaforma (Linux, Windows, macOS) tramite protocollo RFB.

  • VNC integrato in scheda — usa gtk-vnc per mostrare il desktop direttamente in una scheda PCM (richiede gir1.2-gtkvnc-2.0 / gtk-vnc)
  • Client esterno — vncviewer, realvnc-viewer, tigervnc, remmina, krdc
  • Porta default 5900; display :1 = porta 5901, :2 = 5902…
  • Password VNC separata dalla password di sistema

Telnet

Terminale legacy non cifrato. Solo per reti locali sicure o dispositivi che non supportano SSH.

🚫 Non usare Telnet su reti pubbliche — le credenziali sono visibili in chiaro sul traffico di rete.

Mosh Mobile Shell

Alternativa a SSH per connessioni instabili, roaming e alta latenza.

  • Sopravvive a disconnessioni e cambi di rete (WiFi → 4G)
  • Richiede mosh installato sia sul client che sul server
  • Usa UDP per il traffico dati, SSH solo per l'autenticazione iniziale

Seriale Console seriale

Connessione diretta a dispositivi tramite porta RS-232: router, switch, server, Raspberry Pi, Arduino.

  • Dispositivo — es. /dev/ttyUSB0, /dev/ttyACM0, /dev/ttyS0
  • Baud rate — valori comuni: 9600, 115200 (controlla la documentazione del dispositivo)
  • Se il dispositivo non risponde: sudo usermod -a -G dialout $USER (poi rilogin)

Exec Comando locale generico

Esegui qualsiasi comando shell nel terminale embedded di PCM — script, tool locali, monitoraggi live.

  • Il comando viene eseguito localmente sul tuo PC (non su un host remoto)
  • Il campo Comando accetta qualsiasi stringa bash: pipe, redirects, variabili, argomenti
  • Il terminale integrato mostra output e riceve input interattivi (htop, less, python REPL, ecc.)
  • Supporta le variabili globali {NOME}
💡 Esempi: htop, tail -f /var/log/syslog, watch -n1 df -h, python3 ~/scripts/monitor.py
💡 Client e terminali: i combo Client RDP, Client VNC e Terminale esterno mostrano solo i programmi effettivamente installati nel sistema. Se installi un nuovo client dopo aver aperto PCM, riapri il dialog sessione per vederlo comparire. Se un profilo salvato usa un client non più installato, il combo lo mostra con il suffisso ⚠ non trovato.

🔑 Autenticazione

Password

Puoi salvare la password nel profilo o lasciarla vuota — PCM la chiederà con un dialog dedicato a ogni connessione.

⚠️ Le password salvate in connections.json sono in chiaro. Per macchine di produzione o condivise, preferisci le chiavi SSH.

Chiave privata SSH (consigliata)

1
Vai al tab Autenticazione nel dialog sessione. Nella sezione "Gestione chiavi SSH" puoi vedere le chiavi già presenti in ~/.ssh/.
2
Genera nuova chiave: scegli il tipo (ed25519 consigliato), inserisci un commento, clicca ⚙ Genera. La coppia viene salvata in ~/.ssh/.
3
Copia sul server: clicca 📤 Copia chiave pubblica sul server — PCM lancia ssh-copy-id in un xterm visibile dove puoi digitare la password del server (una sola volta).
4
Seleziona la chiave dal combo "Chiavi in ~/.ssh:" — il campo "Chiave privata" si compila automaticamente.
💡 Per non digitare la passphrase ad ogni connessione, usa ssh-agent:
eval $(ssh-agent) && ssh-add ~/.ssh/id_ed25519

Jump Host (Bastion Host)

Per raggiungere server non accessibili direttamente (DMZ, reti private):

  • Tab Autenticazione → sezione Jump Host (Bastion)
  • Jump host — indirizzo del server intermedio
  • Utente jump — utente sul bastion (può differire dall'utente finale)
  • Porta jump — porta SSH del bastion (default 22)

Il risultato è equivalente a ssh -J utente@bastion:porta utente@host-finale, gestito in modo trasparente.

ℹ️ Il Jump Host è disponibile per SSH, SFTP e Mosh.

SSH Agent Forwarding

Propaga le chiavi dell'ssh-agent locale attraverso la connessione SSH, permettendo di autenticarsi su server successivi senza copiare le chiavi private.

Nel dialog sessione SSH, tab ⚙ Avanzate: spunta "Agent Forwarding (-A)" — aggiunge il flag -A al comando SSH.

  • Accesso a server di destinazione tramite bastion host, senza copiare le chiavi sul bastion
  • Deployment da un server CI che deve autenticarsi su altri server
  • Clonare repository Git da un server remoto usando le tue chiavi locali
# Con Agent Forwarding attivo, dal server remoto puoi:
git clone git@github.com:utente/repo.git  # usa le tue chiavi locali
ssh altro-server                           # autenticazione trasparente
⚠️ Abilita l'Agent Forwarding solo su server di cui ti fidi. Un amministratore del server remoto potrebbe usare l'agent locale per autenticarsi su altri server a tuo nome.

📥 Import da altre applicazioni

Da Strumenti → Importa da applicazione esterna importi le connessioni da Remmina, Remote Desktop Manager, PuTTY e dal file ~/.ssh/config senza reinserirle manualmente.

Importa da Remmina

  1. Vai su Strumenti → Importa da applicazione esterna → 🖥 Remmina…
  2. PCM legge la cartella ~/.local/share/remmina/ oppure puoi selezionare una cartella diversa
  3. Verificata l'anteprima (protocollo, host, numero connessioni), conferma l'importazione

Protocolli importati: SSH, SFTP, RDP, VNC, Telnet. I gruppi Remmina diventano gruppi PCM.

Importa da Remote Desktop Manager (XML o JSON)

  1. In RDM: File → Esporta → scegli formato XML (.rdm) o JSON
  2. In PCM: Strumenti → Importa → Remote Desktop Manager (XML)… oppure (JSON)…
  3. Seleziona il file, verifica l'anteprima, conferma
ℹ️ Le password non vengono importate — RDM le cifra con una chiave legata all'account. PCM chiederà le credenziali alla prima connessione.

Importa da PuTTY

PCM legge direttamente le sessioni salvate da PuTTY (Linux/Wine) senza che PuTTY debba essere installato.

  1. In PCM: Strumenti → Importa da applicazione esterna → 🐢 PuTTY…
  2. PCM scansiona automaticamente ~/.putty/sessions/
  3. Le sessioni PuTTY vengono convertite in profili SSH nel gruppo "PuTTY"

Campi importati: host, porta, utente, chiave privata, protocollo (ssh/telnet/serial). Le sessioni con nome "Default Settings" vengono saltate.

Importa da ~/.ssh/config

Converte i blocchi Host del file di configurazione SSH in sessioni PCM.

  1. In PCM: Strumenti → Importa da applicazione esterna → 🔑 SSH Config…
  2. PCM legge ~/.ssh/config (o un file alternativo a tua scelta)
  3. Ogni blocco Host diventa una sessione nel gruppo "SSH Config"

Campi importati: Hostname, Port, User, IdentityFile, ProxyJump (convertito in Jump Host PCM). I wildcard (Host *) vengono ignorati.

💡 Dopo l'importazione da SSH Config, le sessioni PCM supportano tutte le funzioni avanzate (macro, log, WoL, pre-cmd) che ~/.ssh/config non prevede. Puoi arricchire i profili importati dal dialog di modifica.

Importa da MobaXterm

  1. In MobaXterm: Sessions → Export all sessions → salva il file .mobalink o .mxtsessions
  2. In PCM: Strumenti → Importa da applicazione esterna → MobaXterm…
  3. Seleziona il file esportato, verifica l'anteprima (protocollo, host, numero sessioni), conferma

Protocolli importati: SSH, RDP, VNC, Telnet, SFTP. I gruppi MobaXterm diventano gruppi PCM. Le password non vengono importate — MobaXterm le cifra con una chiave di sistema.

⌨ Terminale integrato

PCM usa VTE (Virtual Terminal Emulator) come motore del terminale — il terminale nativo GTK3 usato da GNOME Terminal. Rendering font ottimale, supporto Wayland nativo, nessun layer di compatibilità.

Copia e incolla

  • Copia — seleziona il testo con il mouse (Ctrl+Shift+C per la clipboard)
  • Incolla — Ctrl+Shift+V oppure tasto centrale del mouse
  • Incolla con tasto destro — abilitabile per singola sessione nel tab Terminale

Ricerca nell'output del terminale

Usa Ctrl+F nel terminale attivo per aprire la barra di ricerca. La ricerca scorre il contenuto visibile e il buffer scrollback di VTE.

  • Ricerca in avanti e indietro con ↑ / ↓ o i pulsanti ← →
  • Le corrispondenze vengono evidenziate nel testo
  • Chiudi la barra con Esc

Input password automatico

PCM non usa sshpass per passare le password. Al login, PCM digita direttamente la password nel terminale VTE quando il server la richiede — esattamente come farebbe un utente.

PCM usa due livelli complementari:

  1. Primario — feed_child: PCM monitora l'output del terminale. Quando compare un prompt di autenticazione, invia la password digitandola nel terminale. Funziona con SSH standard, Cisco (enable/secret), PAM, RADIUS, 2FA e qualsiasi prompt personalizzato.
  2. Fallback — SSH_ASKPASS: Per i casi in cui SSH non mostra il prompt nel terminale (autenticazione keyboard-interactive con OpenSSH ≥ 8.4), PCM imposta la variabile SSH_ASKPASS_REQUIRE=force e un piccolo script helper. SSH interroga lo script silenziosamente, senza mostrare nulla a schermo.

Vantaggi rispetto a sshpass

  • La password non appare mai in ps aux o nella command line
  • Nessuna dipendenza da sshpass installato
  • La password non viene passata come argomento a nessun processo esterno

Prompt riconosciuti

  • password: / Password: — SSH standard
  • Password for user@host:
  • Secret: / enable secret: — Cisco
  • Passcode: — RADIUS/MFA
  • Verification code: — Google Auth
  • Enter password: e varianti

Log output del terminale

Abilitando "Registra output su file" nel tab Terminale del profilo, VTE salva automaticamente tutto l'output in un file pcm_YYYYMMDD_HHMMSS.log nella cartella configurata.

💡 Il log è indispensabile per la funzione Esporta comandi.sh: senza log attivo, PCM può tracciare solo i comandi inviati tramite macro e multi-exec, non quelli digitati direttamente in console.

📁 Browser SFTP e FTP

Browser SFTP laterale

Si apre automaticamente accanto al terminale SSH (se l'opzione è attiva nel tab Pannelli). Permette di navigare il filesystem remoto senza aprire un tab separato.

  • Navigazione cartelle con doppio clic
  • Upload e download con barra di progresso in tempo reale (byte trasferiti / totale)
  • Creazione cartelle, rinomina, eliminazione
  • Visualizzazione permessi, dimensioni e date di modifica

Browser SFTP dual-pane (tab dedicato)

Si apre in un tab autonomo aprendo una sessione con protocollo SFTP o via tasto destro → "Apri SFTP/FTP qui".

  • Pannello locale e remoto affiancati (stile WinSCP)
  • Coda trasferimenti con progresso, velocità (byte/s) ed ETA
  • Pausa e annullamento dei trasferimenti in corso
  • Ricorsività su cartelle (upload/download di alberi di directory)
  • Sincronizzazione directory — pulsante Sincronizza nella toolbar (vedi sotto)

Sincronizzazione directory locale ↔ remota

Disponibile nel browser SFTP dual-pane tramite il pulsante Sincronizza nella toolbar.

1
Naviga alla cartella voluta nel pannello locale e in quello remoto.
2
Clicca Sincronizza — PCM scansiona entrambe le cartelle confrontando dimensione e data di modifica.
3
Si apre il dialog con la tabella diff: Solo locale, Solo remoto, Modificato, Uguale. La direzione suggerita (→ remoto / ← locale) è calcolata automaticamente in base alla data più recente.
4
Filtra per stato, seleziona/deseleziona i file, modifica la direzione se necessario, poi clicca Sincronizza selezionati.
5
I trasferimenti vengono accodati e eseguiti con la coda esistente (progresso, velocità, ETA visibili).

Browser FTP/FTPS integrato

Stesso comportamento del browser SFTP dual-pane, ma usa FTP o FTPS:

  • FTP plain (porta 21) o FTPS con TLS esplicito
  • Modalità passiva (PASV) per NAT/firewall

🔀 Gestione Tunnel SSH

L'Indicatore Tunnel nella toolbar mostra a colpo d'occhio il numero di tunnel attivi. Cliccandolo si apre un popup rapido con l'elenco dei tunnel attivi, lo stato, e un pulsante Stop individuale — senza aprire il Gestore completo. L'indicatore si aggiorna ogni 3 secondi e lo stato dei tunnel sopravvive alla chiusura e riapertura del dialog.

Per la gestione completa (aggiungere, modificare, avviare, fermare tutti i tunnel) usa Apri Gestore Tunnel dal popup oppure Strumenti → Gestore Tunnel SSH.

ℹ️ I tunnel SSH si gestiscono esclusivamente dal Gestore Tunnel — non esiste più il protocollo "SSH Tunnel" nella lista sessioni. Questo evita duplicazioni e permette di avviare, fermare e monitorare tutti i tunnel da un solo posto.

Creare un tunnel SOCKS (proxy)

1
Apri il Gestore Tunnel da Toolbar → Tunnel oppure Strumenti → Gestore Tunnel SSH
2
Clicca + Aggiungi, scegli tipo Proxy SOCKS (-D), inserisci SSH Host e porta locale (es. 1080)
3
Clicca ▶ Avvia — il tunnel rimane attivo in background anche dopo aver chiuso il dialog
4
Configura il browser: Proxy → SOCKS5 → 127.0.0.1:1080

Tunnel locale (port forward)

Esempio: accedere a un database MySQL remoto come se fosse locale:

Tipo:          Locale (-L)
Host SSH:      server.example.com   # server con accesso al DB
Porta locale:  13306                # porta libera sul tuo PC
Host remoto:   db-interno           # hostname del DB visto dal server SSH
Porta remota:  3306                 # porta MySQL standard

# Poi connettiti con:
mysql -h 127.0.0.1 -P 13306 -u root

Tipi di tunnel supportati

TipoFlag SSHDescrizione
Locale (-L)-L porta:host:porta_remotaInoltra una porta locale verso un host/porta remoti tramite il server SSH
Remoto (-R)-R porta_remota:host:portaEspone una porta locale sul server remoto (utile per reverse proxy o accesso a macchina NAT)
Proxy SOCKS (-D)-D porta_localeCrea un proxy SOCKS5 locale — tutto il traffico viene instradato attraverso il server SSH

Gestione tunnel

Dal Gestore Tunnel (Strumenti → Gestore Tunnel SSH) puoi:

  • Vedere lo stato di ogni tunnel (attivo / arrestato)
  • Avviare (▶) o fermare (■) singoli tunnel
  • Modificare la configurazione di un tunnel esistente
  • Eliminare tunnel non più necessari

L'Indicatore Tunnel nella toolbar mostra il numero di tunnel attivi. Cliccandolo si apre un popup rapido — senza aprire il Gestore completo. L'indicatore si aggiorna ogni 3 secondi.

Cifratura credenziali

PCM può proteggere utenti e password salvati in connections.json con cifratura AES-256 (Fernet). La chiave viene derivata dalla password master con PBKDF2-HMAC-SHA256 (480.000 iterazioni) e non viene mai scritta su disco.

Al primo avvio assoluto, se connections.json non esiste ancora, PCM chiede se abilitare la protezione delle credenziali. Agli avvii successivi con cifratura attiva, PCM mostra un dialog di sblocco prima di aprire la finestra principale (3 tentativi massimi).

Da Strumenti → Imposta password globale… puoi:

  • Abilitare la cifratura — se non era attiva; tutti i profili vengono immediatamente cifrati
  • Cambiare la password — inserisci quella vecchia e quella nuova; tutti i profili vengono ricifrati
  • Disabilitare la cifratura — inserisci la password attuale; le credenziali tornano in chiaro nel JSON

I campi user e password di ogni profilo in connections.json vengono cifrati con prefisso ENC: seguito dal token Fernet base64. Tutti gli altri campi restano in chiaro.

⚠️ Se dimentichi la password master non è possibile recuperare le credenziali. Tieni una copia sicura della password. Per resettare: elimina connections.json e la sezione crypto da pcm_settings.json.

Wake-on-LAN

PCM può svegliare una macchina spenta prima di connettersi, inviando un magic packet UDP all'indirizzo MAC configurato. La funzione è disponibile per SSH, Mosh, RDP, VNC e Telnet.

Nel dialog di modifica sessione, tab Connessione, sezione Wake-on-LAN:

  • Attiva WoL — spunta per abilitare l'invio automatico del magic packet prima di ogni connessione
  • Indirizzo MAC — indirizzo MAC fisico della scheda di rete della macchina remota (formato AA:BB:CC:DD:EE:FF)
  • Attesa risposta — secondi massimi di attesa prima che PCM tenti la connessione (default 20 s)

All'apertura della sessione PCM costruisce il magic packet e lo invia in broadcast UDP sulla porta 9. Appare una finestra di progresso con ping ripetuti verso l'host; appena l'host risponde la connessione parte automaticamente.

⚠️ Wake-on-LAN funziona solo sulla rete locale o tramite broadcast diretto. Assicurati che la scheda di rete e il BIOS/UEFI della macchina remota abbiano WoL abilitato.

Macro per sessione

Comandi rapidi collegati a una specifica sessione, inviabili al terminale con un singolo clic dal pannello sessioni.

Nel dialog di creazione/modifica sessione, tab ⚡ Macro: aggiungi coppie Nome/Comando. Usa i pulsanti ➕, ✏, 🗑, ▲/▼ per gestire l'ordine.

Per usare una macro: tasto destro sulla sessione nel pannello → ⚡ Macro → seleziona il comando. Il comando viene inviato istantaneamente al terminale attivo della sessione tramite clipboard (nessun ritardo di digitazione).

Esempi di macro utili

  • df -h — spazio disco
  • tail -f /var/log/syslog — log di sistema live
  • sudo systemctl restart nginx — riavvio servizio
  • uptime && free -h — stato sistema
  • docker ps — container attivi

Differenza con il Comando avvio

Il Comando avvio (tab Terminale) viene eseguito automaticamente alla connessione. Le macro vengono eseguite manualmente su richiesta, in qualsiasi momento durante la sessione.

Comando pre-connessione locale

Esegui un comando shell sul tuo PC prima che PCM apra la connessione remota. Ideale per attivare VPN, montare volumi o qualsiasi operazione locale preparatoria.

Nel dialog di modifica sessione, tab Terminale:

  • Cmd locale pre-connessione — comando bash da eseguire prima di connettersi
  • Timeout pre-cmd — secondi massimi di attesa (0 = nessun timeout, default 15 s)

Il comando viene wrappato con il comando di connessione in un unico processo bash. Se il pre-comando esce con codice diverso da zero (errore), la connessione non parte.

bash -c 'pre-cmd && echo "Pre-connessione OK. Connetto..." && ssh utente@host'

Esempi pratici

  • wg-quick up vpn0 — attiva VPN WireGuard
  • openfortivpn --config=/etc/vpn.conf — attiva FortiVPN
  • sudo openvpn --config ~/vpn.ovpn & — avvia OpenVPN
  • sshuttle -r gateway 10.0.0.0/8 & — tunnel rete

Nota sul timeout

Imposta il timeout a un valore ragionevole per il tuo pre-comando. Una VPN che impiega 10 s ad attivarsi richiede un timeout di almeno 15–20 s. Con 0 il pre-comando può girare indefinitamente (utile per comandi in background).

Esporta apri-connessione.sh

Genera uno script bash standalone che apre la connessione senza avere PCM aperto. Utile per condividere o per uso da terminale.

Tasto destro sulla sessione nel pannello → 📄 Esporta apri-connessione.sh…. Si apre un dialog con anteprima dello script (modificabile), pulsante 💾 Salva (con chmod 755 automatico) e 📋 Copia negli appunti.

Lo script include, se configurati: blocco pre-cmd locale, blocco Wake-on-LAN, e il comando di connessione.

Esporta comandi.sh

Genera un replay script con tutti i comandi eseguiti durante la sessione. Il replay si connette via SSH e ri-esegue i comandi in sequenza.

Tasto destro sul tab della sessione attiva (non sul pannello sessioni) → 📄 Esporta comandi.sh….

⚠️ Questa funzione è accessibile dal tab del terminale, non dal pannello sessioni. Il pannello sessioni ha "Esporta apri-connessione.sh" (script di connessione), il tab ha "Esporta comandi.sh" (replay dei comandi).

PCM distingue due scenari:

🔴 Log non abilitato

Appare un avviso con le istruzioni per abilitare il log. È comunque possibile esportare i comandi inviati tramite macro e multi-exec con il pulsante "Esporta solo comandi PCM".

🟢 Log abilitato

PCM parsa il file di log, estrae i comandi digitati riconoscendo le righe che seguono un prompt bash/zsh, e li unisce ai comandi tracciati internamente.

Il dialog di export mostra una tabella con tutti i comandi (modificabili con doppio clic, riordinabili con ▲/▼, eliminabili). L'anteprima dello script si aggiorna in tempo reale. Per sessioni SSH lo script usa un heredoc per eseguire tutti i comandi in una sola connessione.

Port forwarding nella sessione

Le sessioni SSH supportano il port forwarding via flag avanzati nel tab ⚙ Avanzate. Puoi specificare direttamente opzioni -L, -R o -D come argomenti extra SSH, oppure usare il Gestore Tunnel SSH per la gestione persistente.

Audit log

PCM registra automaticamente ogni connessione aperta in un log centralizzato (audit_log.json) con timestamp, durata, protocollo e stato.

Da Strumenti → Audit log: si apre un viewer con tabella delle connessioni (Timestamp, Sessione, Host, Protocollo, Durata, Stato). Azioni disponibili:

  • 🗑 Cancella log — svuota il file audit_log.json

ℹ️ L'audit log non contiene password o credenziali — solo metadati di connessione. È complementare al log output terminale: quello registra il contenuto della sessione, questo i metadati.

Broadcast terminali

Invia lo stesso testo (o comando) a più terminali contemporaneamente — ideale per eseguire operazioni su un cluster di server in parallelo.

Da Strumenti → Broadcast terminali: il dialog elenca tutti i terminali aperti con un checkbox per ciascuno. Scrivi il testo nell'area di testo, scegli se aggiungere automaticamente Invio (default: sì), poi clicca ⚡ Invia.

💡 Il Broadcast funziona solo sui terminali VTE embedded (tab SSH/Telnet/Mosh/Exec aperti in PCM). Le sessioni RDP e VNC in finestra esterna non sono raggiungibili perché non usano il terminale interno.

Differenza con Multi-exec

Multi-exec (Ctrl+Shift+M) usa la clipboard e invia con Invio automatico. Broadcast è più flessibile: puoi inviare testo parziale (senza Invio), più righe, e vedi la selezione dei terminali esplicitamente.

Casi d'uso tipici

  • apt update && apt upgrade -y — aggiorna tutti i server
  • sudo systemctl restart nginx — riavvia su tutto il cluster
  • df -h — controlla spazio su tutti i server

❓ Domande frequenti

Le icone non appaiono — vedo solo pallini colorati

Le icone PNG devono essere presenti nella cartella icons/ accanto a PCM.py. Verifica i file .png. Se hai solo .svg, esegui bash convert_icons.sh dalla cartella di PCM.

La connessione SSH fallisce con "Host key verification failed"

Il server SSH ha cambiato la sua impronta. Rimuovi la vecchia voce:

ssh-keygen -R hostname

Oppure disabilita StrictHostKeyChecking nel profilo (tab Avanzate) o nelle impostazioni globali.

Il terminale non si apre / errore "cannot open display"

La versione GTK3 usa VTE che funziona sia su X11 che su Wayland nativamente. Se si verificano problemi, verifica di essere in una sessione grafica e che le librerie GTK3 siano installate: python3 -c "import gi; gi.require_version('Vte', '2.91')".

RDP si apre ma lo schermo è nero

Prova a cambiare client da xfreerdp3 a xfreerdp nel profilo. Su alcune versioni di Windows Server potrebbe servire aggiungere /sec:rdp tra le opzioni avanzate.

VNC si connette ma lo schermo è nero (specialmente su Windows)

Problema noto con gtk-vnc integrato e server VNC Windows (TightVNC, RealVNC, UltraVNC): il pixel format viene negoziato prima che la depth sia impostata. Suggerimenti:

  • Nel profilo VNC, tab Avanzate, prova la profondità colore 16 bpp invece di 32 bpp
  • Oppure usa il client esterno: in Avanzate seleziona Client esterno e scegli tigervncviewer, vncviewer o un client personalizzato
  • Su server Windows, assicurati che il server VNC supporti l’autenticazione RFB standard (VNC password) e non solo Windows NTLMv2

Client VNC esterno non si apre / errore nel percorso

Se hai aggiunto un client VNC personalizzato in Impostazioni → Strumenti personalizzati, verifica che il Percorso sia corretto e che il file sia eseguibile. Per client nel PATH di sistema basta il nome binario (es. vncviewer).

Lo snippet non appare nel menu contestuale del terminale

Il menu contestuale con “Inserisci snippet…” appare solo se è configurato almeno uno snippet nella Libreria. Vai in Strumenti → Libreria Snippet, aggiungi uno snippet e riprova.

Il browser SFTP non si connette

Il browser SFTP usa paramiko. Verifica l'installazione:
.venv/bin/python3 -c "import paramiko"
Se manca: .venv/bin/pip install paramiko dalla cartella gtk3.

Le macro sono lente / arrivano con ritardo

PCM invia le macro tramite la clipboard GDK (nativa GTK3) in modo istantaneo, indipendentemente dalla lunghezza del comando. Se su sessioni X11 riscontri problemi di clipboard, verifica che xclip sia installato: which xclip. Su Wayland la clipboard è gestita nativamente senza dipendenze esterne.

"Esporta comandi.sh" non trova comandi

I comandi digitati direttamente in xterm vengono catturati solo attraverso il log xterm. Assicurati che nel profilo della sessione sia attivo "Registra output su file" e che la sessione sia stata riconnessa dopo aver abilitato il log. I comandi inviati tramite macro e multi-exec sono sempre tracciati indipendentemente dal log.

Wake-on-LAN non sveglia la macchina

Verifica: (1) WoL abilitato nel BIOS/UEFI della macchina remota; (2) WoL abilitato sulla scheda di rete — ethtool eth0 | grep Wake; (3) la macchina è sulla stessa rete locale o il router permette il broadcast diretto; (4) l'indirizzo MAC è corretto.

Come faccio a connettere a un server con porta SSH non standard?

Specifica la porta nel campo "Porta" del dialog sessione oppure usa il formato utente@host:2222 direttamente nella barra Quick Connect.

Posso avere PCM su più macchine con le stesse sessioni?

Sì — il file connections.json è portabile. Sincronizzalo con Dropbox, Nextcloud, Git o copialo manualmente. Usa File → Esporta sessioni per backup e trasferimenti.

I tunnel SSH si chiudono quando chiudo PCM?

Se minimizzi nel system tray i tunnel rimangono attivi. Se esci definitivamente, tutti i processi SSH vengono terminati. Per tunnel permanenti, usa un systemd user service o screen/tmux sul server.

Dove sono salvate le impostazioni?

  • connections.json — tutte le sessioni (in chiaro, JSON modificabile)
  • pcm_settings.json — impostazioni globali, scorciatoie, sessioni recenti
  • audit_log.json — log audit connessioni (metadata, no password)
  • /tmp/pcm_logs/ — log dei terminali (se abilitato, percorso configurabile)

KeePassXC non viene trovato / "non in esecuzione"

Verifica: (1) KeePassXC è aperto e il database è sbloccato; (2) Browser Integration è abilitata in KeePassXC → Strumenti → Impostazioni → Browser Integration; (3) il socket Unix esiste: ls /run/user/$(id -u)/org.keepassxc.KeePassXC.BrowserServer. Usa Strumenti → KeePassXC per il test di connessione.

Il Broadcast non raggiunge tutti i terminali

Il Broadcast funziona solo sui terminali VTE embedded (tab SSH/Telnet/Mosh/Exec aperti in PCM). Le sessioni RDP e VNC in finestra esterna non sono raggiungibili perché non usano il terminale interno.

Il pannello Monitor SSH non appare

Verifica che nella sessione (tasto destro → Modifica → tab Pannelli) sia abilitata almeno una delle sezioni monitor (CPU/RAM, Processi, Disco, Rete). Se nessuna sezione è attiva il pannello rimane nascosto. Riconnetti la sessione dopo la modifica.

Il pannello Monitor mostra zero o nessun dato

Il pannello si connette via SSH (paramiko) all'host della sessione. Verifica: (1) l'host è raggiungibile direttamente da Python senza tramite un jump host; (2) la chiave host è in ~/.ssh/known_hosts; (3) se la sessione usa un jump host, il pannello non può passarci — usa il monitor standalone (tasto destro → Monitor sistema…) oppure apri un tunnel manuale.

Il tab Logs nel pannello non mostra nulla / rimane come placeholder

Il tab Logs appare automaticamente dopo che la connessione SSH del monitor è stabilita (circa 5-15 secondi). Se persiste il placeholder, verifica lo stato della connessione monitor nella statusbar del pannello. Clicca ▶ Avvia per (ri)avviare lo streaming.

Il pulsante Sincronizza nel browser SFTP non c'è

Il pulsante "Sincronizza" è disponibile solo nel browser dual-pane SFTP (tab aperto tramite sessione SFTP o tasto destro → "Apri SFTP/FTP qui"), non nel browser SFTP laterale. Assicurati di avere entrambi i pannelli (locale e remoto) navigati nella cartella voluta prima di sincronizzare.

Il ripristino sessioni all'avvio non funziona

Verifica: (1) l'opzione è abilitata in Impostazioni → Generale; (2) la sessione è ancora presente in connections.json con host, protocollo, porta e utente identici a quelli della sessione aperta al momento della chiusura. Le sessioni aperte via Quick Connect (senza profilo salvato) non vengono ripristinate.

⌨ Avvio dalla riga di comando

PCM supporta l'apertura di connessioni direttamente dalla riga di comando tramite URI. Se PCM è già in esecuzione, la connessione viene aggiunta come nuova tab nella finestra esistente, senza chiedere nuovamente la password master se le credenziali sono già sbloccate.

ℹ️ PCM usa D-Bus per rilevare se è già aperto. Un secondo invocation invia l'URI all'istanza esistente via D-Bus e termina subito — non si aprono due finestre PCM.

Formato URI

python3 PCM.py  proto://[utente@]host[:porta][?opzioni]
ProtocolloSchema URIPorta default
SSHssh://22
Telnettelnet://23
RDPrdp://3389
VNCvnc://5900
SFTPsftp://22
FTPftp://21
FTPSftps://21
Moshmosh://60001

Opzioni query string

Solo per connessioni ad-hoc (host non presente tra le sessioni salvate):

ParametroValoriDescrizione
modeinternal / externalForza apertura embedded nel tab PCM oppure in finestra/client esterno
terminalxterm, kitty, …Terminale esterno specifico da usare (solo SSH/Telnet con mode=external)

Ricerca sessione salvata

Quando si passa un URI, PCM cerca la sessione nell'ordine seguente:

  1. Nome sessione uguale all'host nell'URI (confronto case-insensitive)
  2. Hostname della sessione uguale all'host nell'URI, con lo stesso protocollo
  3. Hostname della sessione uguale all'host nell'URI, qualsiasi protocollo
  4. Nessuna corrispondenza → connessione ad-hoc con i parametri dell'URI

Se viene trovata una sessione salvata, vengono usate tutte le sue impostazioni (password, chiave, macro, ecc.). È possibile sovrascrivere utente e porta nell'URI.

Esempi — Sessioni salvate

# Apre la sessione di nome "jiraapp" (o con hostname "jiraapp")
python3 PCM.py ssh://jiraapp

# Apre la sessione RDP salvata "windowsserver"
python3 PCM.py rdp://windowsserver

# Apre la sessione SFTP salvata "nas"
python3 PCM.py sftp://nas

# Apre la sessione Telnet — se PCM è già aperto, aggiunge un tab
python3 PCM.py telnet://router-core

# Sovrascrive l'utente della sessione salvata "webserver"
python3 PCM.py ssh://root@webserver

Esempi — Connessioni ad-hoc (non salvate)

# SSH con utente e host
python3 PCM.py ssh://admin@192.168.1.10

# SSH con porta non standard
python3 PCM.py ssh://deploy@prodserver.example.com:2222

# SSH nel terminale interno (default)
python3 PCM.py ssh://user@host

# SSH in terminale esterno
python3 PCM.py 'ssh://user@host?mode=external'

# SSH in terminale esterno specifico
python3 PCM.py 'ssh://user@host?mode=external&terminal=kitty'

# SSH in terminale esterno specifico (altro esempio)
python3 PCM.py 'ssh://user@host?mode=external&terminal=xterm'

# RDP ad-hoc: si apre xfreerdp (default esterno)
python3 PCM.py rdp://192.168.1.50

# RDP con utente, embedded nel pannello PCM
python3 PCM.py 'rdp://administrator@192.168.1.50?mode=internal'

# VNC viewer esterno
python3 PCM.py vnc://raspberrypi

# VNC viewer embedded con porta
python3 PCM.py 'vnc://192.168.1.60:5901?mode=internal'

# SFTP ad-hoc con utente
python3 PCM.py sftp://backup@nas.lan

# FTP ad-hoc
python3 PCM.py ftp://ftp.example.com

# FTPS ad-hoc
python3 PCM.py ftps://secure-ftp.example.com

# Telnet con porta personalizzata
python3 PCM.py telnet://cisco-switch:2323

# Mosh ad-hoc
python3 PCM.py mosh://user@remotehost

Alias consigliati in ~/.bashrc

# Connessione rapida agli host più usati
alias jira='python3 ~/PCM/gtk3/PCM.py ssh://jiraapp'
alias nas='python3 ~/PCM/gtk3/PCM.py sftp://nas'
alias vpn-gw='python3 ~/PCM/gtk3/PCM.py ssh://vpn-gateway'

# RDP al PC Windows
alias winpc='python3 ~/PCM/gtk3/PCM.py rdp://workpc'

# Funzione generica: apri qualsiasi host SSH in PCM
pcm() { python3 ~/PCM/gtk3/PCM.py "ssh://$1"; }

Comportamento con credenziali cifrate

PCM non è aperto

PCM si avvia normalmente, mostra il dialog di sblocco password master, e dopo l'unlock apre automaticamente la connessione richiesta. Non è necessario rilanciare il comando.

PCM è già aperto e sbloccato

La connessione viene aperta immediatamente come nuova tab. Nessuna richiesta di password master.

💡 Ricorda di avviare almeno una volta PCM con il nuovo codice prima di usare la funzione URI: la prima apertura registra il flag D-Bus necessario per il passaggio dell'URI all'istanza esistente.
