Toolbar

The toolbar below the menu bar provides quick access to the main functions:

  • New Session — opens the dialog to create a new connection
  • Local Terminal — opens a local terminal tab
  • Tunnel Indicator — shows the number of active SSH tunnels; click to open a popup listing each active tunnel with an individual Stop button; the indicator refreshes automatically every 3 seconds. The full Tunnel Manager is accessible from the popup or via Tools › SSH Tunnels…

  • Split ▼ — split the terminal area vertically or horizontally

Quick Connect

The bar below the toolbar allows you to quickly connect without saving the session:

  • Select protocol from the drop-down menu (SSH, Telnet, Mosh, Serial…)
  • Enter user@host:port (port is optional, uses default)
  • Press Enter or click ▶ to connect

Tabs (Notebook)

Each active connection opens in a separate tab. You can:

  • Drag tabs to reorder them
  • Right-click on a tab to access the context menu:
    • Move to other panel — move tab to the split panel
    • Open file transfer here — opens SFTP/FTP reusing session credentials
    • ⧉ Duplicate connection — opens a second session to the same host
    • ✖ Close tab — closes the session
  • Ctrl+W — close current tab

Split panels

The terminal area can be split into two panels (vertical or horizontal) via Split ▼ in the toolbar or via View › Split. Drag a tab into the second panel to work on two sessions simultaneously.

  • Single view Ctrl+Alt+1
  • Vertical split Ctrl+Alt+2 — two panels side by side
  • Horizontal split Ctrl+Alt+3 — two panels stacked

System Tray

With active sessions, closing the window minimizes PCM to the system tray instead of terminating connections.

  • Double-click on the icon — reopens the main window
  • Right-click on the icon — menu with quick access to the last 8 used sessions and Quit option

📋 Session Management

Create a new session

Click ➕ in the sidebar, or Ctrl+Shift+N. The dialog contains:

1
Assign a Name (e.g. Production server) and an optional Group (e.g. Work).
2
Choose the Protocol — the dialog adapts showing only relevant fields.
3
Connection tab: Host, Port, User, and protocol-specific options.
4
Authentication tab: password or SSH private key. You can leave blank — PCM will ask at connection time.
5
Terminal tab: color theme, font, size, startup command, local pre-connection command, output log.
6
⚙ Advanced tab: keepalive, X11 forwarding, compression, jump host, open mode, external terminal selector.
7
⚡ Macros tab: define quick commands sendable with one click from the session panel.
8
📊 Panels tab (SSH, VNC Linux, RDP Windows): enables the right info panel and SFTP side browser. For SSH and VNC Linux: CPU/RAM, processes, disk, network, log streaming. For RDP Windows: PowerShell-based panel via OpenSSH. Each section is individually activatable.
9
📝 Notes tab: free text field for annotations, procedures, useful links.
10
Click OK — the session is saved and the connection opens immediately.

Groups

Sessions are organized in groups (folders). You can:

  • Create nested subgroups
  • Expand/collapse groups with a click
  • Move sessions via drag & drop between groups
  • Sort sessions alphabetically within a group (right-click menu)

Session context menu (right-click)

  • ▶ Connect — opens the connection in a new tab
  • ✏ Edit — reopens the dialog with pre-filled data
  • 📋 Duplicate — creates a copy with "(copy)" suffix
  • 📋 Copy command — copies the equivalent shell command to clipboard
  • 📄 Export open-connection.sh… — generates a standalone bash script to open the connection
  • 📂 Open SFTP/FTP here — opens a file transfer browser for the selected SSH/Telnet/Mosh session
  • View log… — opens a streaming log tab (journalctl/tail -f) for the selected SSH session
  • System monitor… — opens a resource monitoring tab (CPU, RAM, disk, network) for the selected SSH session
  • Ping — checks TCP reachability of the host on the configured port
  • ⚡ Macros — submenu with macros defined for this session
  • 🗑 Delete — requires confirmation

Import and export sessions

From File → Export sessions save all sessions to a portable JSON file. From File → Import sessions load a previously exported JSON file. Sessions are saved in connections.json in the PCM folder — editable with any text editor.

🔌 Supported Protocols

SSH Secure Shell

Encrypted remote terminal. The most common protocol for Linux/Unix server management.

  • Authentication: password, SSH key (RSA/Ed25519/ECDSA), SSH agent
  • Supports proxy jump (ProxyJump), port forwarding, X11 forwarding
  • Integrated VTE terminal: scrollback, copy/paste, clickable URLs
  • SFTP browser panel alongside the terminal (enable in session settings)
  • Pre-command and Wake-on-LAN before connection

SFTP SSH File Transfer Protocol

Secure file transfer over an SSH channel. Not FTP — uses an encrypted SSH connection.

  • Same parameters as SSH (host, port, user, key/password)
  • Integrated graphical browser (WinSCP-style): navigate, upload, download, rename, delete
  • Permissions, sizes, and modification dates displayed
  • Open mode — internal browser, SFTP CLI terminal, system file manager

FTP / FTPS File Transfer Protocol

Classic file transfer protocol. FTPS adds TLS encryption over FTP.

  • Supports plain FTP and FTPS (explicit/implicit)
  • Integrated graphical browser similar to the SFTP browser
  • Passive/active mode configurable
  • Anonymous login support

RDP Remote Desktop Protocol

Microsoft remote desktop protocol. Used mainly for Windows machines.

  • Uses xfreerdp (FreeRDP3 recommended)
  • External mode: opens xfreerdp in its own window
  • Internal mode: embeds the RDP session in a PCM tab
  • Resolution, color depth, multi-monitor, audio redirection configurable
  • Drive, clipboard, printer sharing

VNC Virtual Network Computing

Remote desktop protocol, cross-platform (Linux, macOS, Windows).

  • External mode: uses tigervnc, gvncviewer, remmina or any configured client
  • Internal mode: integrated noVNC viewer (WebKit) inside a PCM tab
  • Quality, color depth, scaling configurable for the integrated viewer
  • Password save option in internal mode

Telnet

Unencrypted text protocol. Use only on trusted networks or for legacy systems.

  • Uses system telnet or netcat
  • Integrated VTE terminal identical to SSH

Mosh Mobile Shell

SSH-based protocol resistant to network interruptions. Ideal for mobile or unstable connections.

  • Requires mosh installed on both client and server
  • Maintains the session when IP changes or the laptop sleeps
  • SSH key authentication supported

Serial

Serial console connection via USB/RS-232 adapter.

  • Uses minicom, picocom or custom command
  • Port (/dev/ttyUSB0, /dev/ttyACM0…), baud rate, flow control configurable
  • Integrated VTE terminal

⚡ Exec

Runs any local command in a PCM tab.

  • Command configurable freely (e.g. htop, journalctl -f, scripts)
  • Integrated VTE terminal

🔑 Authentication

Password

You can save the password in the profile or leave it blank — PCM will ask with a dedicated dialog at each connection.

⚠️ Passwords saved in connections.json are stored in plaintext. For production or shared machines, prefer SSH keys.

SSH Private Key (recommended)

1
Go to the Authentication tab in the session dialog. In the "SSH Key Management" section you can see keys already present in ~/.ssh/.
2
Generate new key: choose the type (ed25519 recommended), enter a comment, click ⚙ Generate. The key pair is saved in ~/.ssh/.
3
Copy to server: click 📤 Copy public key to server — PCM runs ssh-copy-id in a visible terminal where you can type the server password (once only).
4
Select the key from the "Keys in ~/.ssh:" combo — the "Private key" field fills automatically.
💡 To avoid typing the passphrase on every connection, use ssh-agent:
eval $(ssh-agent) && ssh-add ~/.ssh/id_ed25519

Jump Host (Bastion Host)

To reach servers not directly accessible (DMZ, private networks):

  • Authentication tab → Jump Host (Bastion) section
  • Jump host — address of the intermediate server
  • Jump user — user on the bastion (may differ from the final user)
  • Jump port — SSH port of the bastion (default 22)

Equivalent to ssh -J user@bastion:port user@final-host, handled transparently.

ℹ️ Jump Host is available for SSH, SFTP and Mosh.

SSH Agent Forwarding

Propagates your local ssh-agent keys through the SSH connection, allowing authentication on subsequent servers without copying private keys.

In the SSH session dialog, ⚙ Advanced tab: tick "Agent Forwarding (-A)" — adds the -A flag to the SSH command.

  • Access destination servers through a bastion, without copying keys to the bastion
  • Clone Git repositories on a remote server using your local keys
⚠️ Enable Agent Forwarding only on servers you trust. A server administrator could use your local agent to authenticate on other servers on your behalf.

📥 Import from External Applications

Import from SSH config (~/.ssh/config)

  1. In PCM: Tools › Import › SSH Config…
  2. PCM reads ~/.ssh/config and creates a profile for each Host entry
  3. Group, keys, port, and proxy settings are preserved

Import from Remmina

  1. In PCM: Tools › Import › Remmina…
  2. PCM scans ~/.local/share/remmina/ for .remmina files
  3. Preview imported sessions, confirm to add

Imported protocols: SSH, SFTP, RDP, VNC, Telnet. Remmina groups become PCM groups.

Import from Remote Desktop Manager (XML or JSON)

  1. In RDM: File → Export → choose XML (.rdm) or JSON format
  2. In PCM: Tools › Import › Remote Desktop Manager (XML)… or (JSON)…
  3. Select the file, check the preview, confirm
ℹ️ Passwords are not imported — RDM encrypts them with an account-bound key. PCM will ask for credentials on first connection.

Import from PuTTY

PCM reads PuTTY saved sessions (Linux/Wine) directly without PuTTY being installed.

  1. In PCM: Tools › Import from external application › 🐢 PuTTY…
  2. PCM automatically scans ~/.putty/sessions/
  3. PuTTY sessions are converted to SSH profiles in the "PuTTY" group

Import from MobaXterm

  1. In MobaXterm: Sessions → Export sessions → save .mxtsessions file
  2. In PCM: Tools › Import › MobaXterm…
  3. Select the file and confirm

PCM automatically inputs the password when the SSH/Telnet prompt appears in the terminal (feed_child mechanism). For SSH ≥ 8.4 it also uses SSH_ASKPASS_REQUIRE=force for silent authentication.

📁 SFTP/FTP File Manager

SFTP sidebar panel

Enable in the session dialog → Panels tab → SFTP side panel. This adds a compact file browser panel alongside the SSH terminal tab without opening a separate tab.

  • Upload and download with real-time progress bar (bytes transferred / total)
  • Create folders, rename, delete
  • Permissions, sizes and modification dates

Dual-pane SFTP browser (dedicated tab)

Opens as its own tab when using the SFTP protocol or via right-click → Open file transfer here on an SSH tab.

  • Side-by-side local and remote panels (WinSCP style)
  • Transfer queue with progress, speed (bytes/s) and ETA per file
  • Pause and cancel in-progress transfers
  • Recursive upload/download of entire directory trees
  • Directory synchronization — Synchronize button in the toolbar (see below)

Directory synchronization local ↔ remote

Available in the dual-pane SFTP browser via the Synchronize button.

1
Navigate to the desired folder in both the local and remote panels.
2
Click Synchronize — PCM scans both folders comparing size and modification time.
3
The diff table opens: Local only, Remote only, Modified, Equal. The suggested direction (→ remote / ← local) is computed from the more recent modification date.
4
Filter by status, select/deselect files, adjust direction as needed, then click Sync selected.
5
Transfers are queued and executed with the existing transfer queue (progress, speed, ETA visible).

FTP/FTPS integrated browser

Same behavior as the dual-pane SFTP browser, but uses FTP or FTPS:

  • Plain FTP (port 21) or FTPS with explicit TLS
  • Passive mode (PASV) for NAT/firewall

⚙ Advanced Features

Pre-command

In the session dialog, Advanced tab → Pre-command: a command run locally before the connection. Useful for:

  • Starting a VPN (nmcli con up vpn-office)
  • Mounting a network share
  • Any preparation script

You can set a timeout after which PCM proceeds with connection even if the command has not terminated.

Port forwarding in session

In the session dialog, Advanced tab → Port Forwarding: define one or more -L/-R rules active for the duration of the session.

External clients

For each protocol you can choose the terminal emulator or external client used to open the connection:

  • Internal terminal (default): VTE embedded in PCM
  • External terminal: xterm, GNOME Terminal, Konsole, Alacritty, Kitty, etc.

The external terminal list is populated automatically based on the emulators installed on the system.

Sessions with unavailable client

If a saved profile uses a client that is no longer installed, the combo shows the entry with the ⚠ not found suffix. Update the profile choosing an available client before connecting.

💡 Use Tools → Check dependencies to see the full list of all clients and terminals detected on the system. The check also runs automatically at startup.

Credential encryption

PCM can protect users and passwords saved in connections.json with AES-256 encryption (Fernet). The key is derived from the master password using PBKDF2-HMAC-SHA256 (480,000 iterations) and is never written to disk.

At the very first launch, if connections.json does not yet exist, PCM asks whether to enable credential protection. On subsequent launches with encryption active, PCM shows an unlock dialog before opening the main window (3 attempts maximum).

From Tools → Set global password… you can:

  • Enable encryption — if not already active; all profiles are immediately encrypted
  • Change the password — enter the old one and the new one; all profiles are re-encrypted
  • Disable encryption — enter the current password; credentials return to plaintext in JSON

The user and password fields of each profile in connections.json are encrypted with the ENC: prefix followed by a Fernet base64 token. All other fields remain in plaintext.

⚠️ If you forget the master password, credentials cannot be recovered. Keep a secure copy. To reset: delete connections.json and the crypto section from pcm_settings.json.

Wake-on-LAN

PCM can wake a powered-off machine before connecting by sending a UDP magic packet to the configured MAC address. Available for SSH, Mosh, RDP, VNC and Telnet.

In the session edit dialog, Connection tab, Wake-on-LAN section:

  • Enable WoL — tick to enable automatic magic packet sending before each connection
  • MAC address — physical MAC address of the remote machine's network card (format AA:BB:CC:DD:EE:FF)
  • Wait time — maximum seconds to wait before PCM attempts the connection (default 20 s)

On session open, PCM builds the magic packet and sends it as UDP broadcast on port 9. A progress window with repeated pings appears; as soon as the host responds the connection starts automatically.

⚠️ Wake-on-LAN only works on the local network or via directed broadcast. Make sure the remote machine's network card and BIOS/UEFI have WoL enabled.

Session macros

Quick commands linked to a specific session, sendable to the terminal with a single click from the session panel.

In the session create/edit dialog, ⚡ Macros tab: add Name/Command pairs. Use the ➕, ✏, 🗑, ▲/▼ buttons to manage order.

To use a macro: right-click on the session in the panel → ⚡ Macros → select the command. The command is sent instantly to the active session terminal via clipboard (no typing delay).

Useful macro examples

  • df -h — disk space
  • tail -f /var/log/syslog — live system log
  • sudo systemctl restart nginx — restart service
  • uptime && free -h — system status
  • docker ps — active containers

Difference with startup command

The startup command (Terminal tab) runs automatically on connection. Macros run manually on demand, at any time during the session.

Local pre-connection command

Run a shell command on your PC before PCM opens the remote connection. Ideal for activating VPN, mounting volumes, or any preparatory local operation.

In the session edit dialog, Terminal tab:

  • Local pre-connection cmd — bash command to run before connecting
  • Pre-cmd timeout — maximum wait seconds (0 = no timeout, default 15 s)

The command is wrapped with the connection command in a single bash process. If the pre-command exits with a non-zero code (error), the connection does not start.

bash -c 'pre-cmd && echo "Pre-connection OK. Connecting..." && ssh user@host'

Practical examples

  • wg-quick up vpn0 — activate WireGuard VPN
  • openfortivpn --config=/etc/vpn.conf — activate FortiVPN
  • sudo openvpn --config ~/vpn.ovpn & — start OpenVPN
  • sshuttle -r gateway 10.0.0.0/8 & — network tunnel

Timeout note

Set the timeout to a reasonable value for your pre-command. A VPN that takes 10 s to activate needs a timeout of at least 15–20 s. With 0 the pre-command can run indefinitely (useful for background commands).

Export open-connection.sh

Generates a standalone bash script that opens the connection without PCM being open. Useful for sharing or terminal use.

Right-click on the session in the panel → 📄 Export open-connection.sh…. A dialog opens with a script preview (editable), a 💾 Save button (with automatic chmod 755) and 📋 Copy to clipboard.

The script includes, if configured: local pre-cmd block, Wake-on-LAN block, and the connection command.

Export commands.sh

Generates a replay script with all commands executed during the session. The replay connects via SSH and re-executes the commands in sequence.

Right-click on the active session tab (not the session panel) → 📄 Export commands.sh….

⚠️ This function is accessible from the terminal tab, not from the session panel. The session panel has "Export open-connection.sh" (connection script), the tab has "Export commands.sh" (command replay).

PCM distinguishes two scenarios:

🔴 Log not enabled

A warning appears with instructions to enable the log. You can still export commands sent via macros and multi-exec using the "Export PCM commands only" button.

🟢 Log enabled

PCM parses the log file, extracts typed commands recognizing lines following a bash/zsh prompt, and combines them with internally tracked commands.

The export dialog shows a table with all commands (editable with double-click, reorderable with ▲/▼, deletable). The script preview updates in real time. For SSH sessions the script uses a heredoc to run all commands in a single connection.

Global variables

From Tools → Global variables (Ctrl+Shift+G). Define NAME = value pairs and use them with the syntax {NAME} in session commands:

# Global variable:
DEPLOY_PATH = /var/www/html/app

# Session "Startup command" field:
cd {DEPLOY_PATH} && git pull && npm run build

Protected mode

From Tools → Protected mode (Ctrl+Shift+P). When active, passwords visible in the interface are replaced with ****. Useful during screen sharing or presentations. Does not modify saved data.

Audit log

PCM automatically records every opened connection in a centralized log (audit_log.json) with timestamp, duration, protocol and status.

From Tools → Audit log: a viewer opens with a connection table (Timestamp, Session, Host, Protocol, Duration, Status). Available actions:

  • 🗑 Clear log — empties the audit_log.json file
  • 📥 Export CSV — saves a copy of the log in CSV format (compatible with Excel, LibreOffice Calc)
ℹ️ The audit log does not contain passwords or credentials — only connection metadata. It is complementary to the terminal output log: the latter records the session content, this one records metadata.

KeePassXC integration

PCM can look up credentials directly in the open KeePassXC database, via the KeePassXC Browser v2 protocol (local Unix socket — no browser extension needed).

Prerequisites: KeePassXC running with unlocked database + Browser Integration enabled (KeePassXC → Tools → Settings → Browser Integration).

1
In the session dialog, Connection tab, click the 🔑 KeePassXC button
2
The search dialog opens: type the server hostname or any keyword
3
Click Search — PCM queries KeePassXC and shows found entries (title, user, URL)
4
Select the desired entry and click Use — user and password are filled in automatically

From Tools → KeePassXC you can check the connection status and the detected socket path.

Broadcast terminals

Send the same text (or command) to multiple terminals simultaneously — ideal for running operations on a cluster of servers in parallel.

From Tools → Broadcast terminals: the dialog lists all open terminals with a checkbox for each. Write the text in the text area, choose whether to automatically add Enter (default: yes), then click ⚡ Send.

💡 Broadcast only works on VTE embedded terminals (SSH/Telnet/Mosh/Exec tabs open in PCM). RDP and VNC sessions in external windows are not reachable because they don't use the internal terminal.

Difference with Multi-exec

Multi-exec (Ctrl+Shift+M) uses the clipboard and sends with automatic Enter. Broadcast is more flexible: you can send partial text (without Enter), multiple lines, and see the terminal selection explicitly.

Typical use cases

  • apt update && apt upgrade -y — update all servers
  • sudo systemctl restart nginx — restart on entire cluster
  • df -h — check space on all servers

Reachability check

Right-click on any session in the panel → 🌐 Check reachability…

A dialog opens that automatically runs three tests in sequence:

TestWhat it checksResult
1. DNSResolves the hostname to an IP address✅ IP resolved or ❌ name not found
2. PingSends 3 ICMP packets, shows RTT and loss✅ Ping OK or ⚠ ICMP blocked
3. TCP portOpens a TCP connection on the protocol port✅ Port open or ❌ timeout
ℹ️ A blocked ping (ICMP filtered by firewall) does not mean the service is unreachable — the TCP port test is the definitive one.

Session restore on startup

PCM can save the list of open sessions on exit and reopen them automatically on the next startup.

From Settings → General → tick "Restore open sessions on next startup".

  • On exit PCM saves host, protocol, port and user for each open tab in pcm_settings.json
  • On next startup it looks up each saved session in connections.json matching those four fields
  • Sessions no longer in connections.json are silently ignored
  • If credentials are encrypted with a master password, restore happens after unlocking

🛠 Settings

Open Settings via the toolbar or File › Settings…. The dialog is divided into sections:

General

  • Language — interface language (Italian, English, German, French, Spanish). Requires restart.
  • Confirm on exit — ask for confirmation before closing with active sessions
  • Default editor — text editor for opening config files
  • Restore open sessions on next startup — saves the list of open tabs on exit and reopens them automatically at the next startup. Requires sessions to be present in connections.json with matching host, protocol, port and user.

Shortcuts

Customize keyboard shortcuts for all PCM actions. Changes take effect immediately without restart.

Tools

PCM auto-detects the paths of external clients (xfreerdp, tigervnc, mosh, etc.). Use Tools › Check dependencies to verify detection and manually override paths if needed.

ℹ️ Terminal settings (theme, font, scrollback, bell) and SSH options are now configured per session in the session dialog — Connection and Terminal tabs. There are no longer global terminal or SSH settings.

📊 SSH Session Info Panel

For any SSH session you can enable a persistent right sidebar that shows server resources in real time, organized in two tabs: Monitor and Logs.

ℹ️ The panel reuses the existing SSH connection opened by the monitor — no extra TCP connection. Credentials are the same as the session.

Enable the panel

In the session edit dialog → Panels tab. Each section is independent:

SectionWhat it shows
CPU and MemoryUsage bars with absolute values (MB/GB), updated every 2 seconds
ProcessesSortable table (click any column header) with PID, CPU%, Mem%, Command — ✖ kill button per row
Disk UsageOne card per partition: mount point, device, used/total, free space, color-coded progress bar
Network UsageDownload/upload speed in bytes/s with dual-line sparkline (blue = download, red = upload); network interface selector
Logs (Logs tab)Streaming journalctl or tail -f with regex filter, level colouring and auto-scroll

Monitor tab

The panel updates every 2 seconds via a persistent remote loop (a single SSH channel stays open for the entire session duration). No new channel is opened per refresh — no "no more sessions" risk.

Sort processes

Click any column header (PID, CPU%, Mem%, Command) to sort the table. Click again to reverse. Default: sorted by CPU% descending.

Kill a process

Click the ✖ button on the process row — PCM asks for confirmation, then sends kill -15 <pid> (with kill -9 as fallback).

Network interface selector

The combo at the top of the Network section is populated automatically with the available interfaces from the first read of /proc/net/dev. Select the interface to monitor (e.g. enp2s0, wlan0).

Logs tab

The Logs tab appears automatically when the monitor SSH connection is established. It shows a streaming log viewer directly in the right panel.

SourceCommand executed
journalctl (all)journalctl -f -n 200
journalctl (errors)journalctl -f -n 200 -p err..emerg
journalctl -u <service>journalctl -f -n 200 -u service.name
tail -f <file>tail -f /path/to/file
  • Regex filter — show only lines matching the regular expression
  • Automatic colouring — red for error/fail/crit, yellow for warn/notice
  • Match highlighting — the parts matching the filter are highlighted in green
  • Auto-scroll — follows the output in real time (can be disabled)
  • ▶ Start — restarts streaming (change source, reinitialize the channel)

Resize the panel

The separator between the terminal area and the info panel is draggable. The panel has a minimum width of 240 px and can be widened freely.

❓ FAQ

PCM doesn't start — missing module error

Run bash setup.sh from the project folder to install all dependencies. If the error persists, check that the virtual environment is activated:

source gtk3/.venv/bin/activate
python3 gtk3/PCM.py

SSH connection fails with "Host key verification failed"

The server's host key has changed. To fix:

ssh-keygen -R hostname

Then reconnect — SSH will save the new key.

GTK3 / GObject not accessible from Python

Install the Python GI bindings:

sudo apt-get install python3-gi python3-gi-cairo gir1.2-gtk-3.0

If using a virtual environment, make sure it was created with --system-site-packages:

python3 -m venv --system-site-packages gtk3/.venv

xfreerdp not found

Install FreeRDP3:

# Debian/Ubuntu
sudo apt-get install freerdp3-x11

# Fedora
sudo dnf install freerdp

# Arch
sudo pacman -S freerdp

VNC exits immediately

Check the log at /tmp/pcm_vnc.log for details. Common causes:

  • VNC client not installed (tigervnc, gvncviewer)
  • Wrong port (default VNC is 5900)
  • Server requires authentication type not supported by the chosen client

WoL doesn't work

Check that:

  • wakeonlan or wol is installed
  • The MAC address is correct
  • WoL is enabled in the machine's BIOS/UEFI
  • The network adapter supports WoL (check with ethtool eth0 | grep Wake)
  • The machine and PCM are on the same subnet (or configure directed broadcast)

How do I update PCM?

Run bash setup.sh from the project folder — if PCM is already installed, the script offers to download updates via git pull.

Where are settings and sessions saved?

  • connections.json — sessions (project root)
  • pcm_settings.json — settings (project root)
  • ~/.local/share/applications/pcm.desktop — desktop launcher

How do I use a SOCKS proxy for browsing?

  1. Create a Dynamic (SOCKS) tunnel: New Tunnel → type: Dynamic → local port: 1080
  2. In Firefox/Chromium → Proxy settings → SOCKS5 → 127.0.0.1:1080

The SSH monitor panel doesn't appear

Make sure at least one monitor section is enabled in the session (right-click → Edit → Panels tab): CPU/RAM, Processes, Disk, or Network. If no section is active the panel stays hidden. Reconnect the session after changing the setting.

The monitor panel shows zero or no data

The panel connects via SSH (paramiko) directly to the session host. Check: (1) the host is directly reachable from Python without a jump host; (2) the host key is in ~/.ssh/known_hosts; (3) if the session uses a jump host, the panel cannot route through it — use the standalone monitor instead (right-click → System monitor…) or set up a manual tunnel.

The Logs tab in the panel stays as a placeholder

The Logs tab appears automatically after the monitor SSH connection is established (5–15 seconds). If the placeholder persists, check the monitor connection status in the panel's status bar. Click ▶ Start to (re)start streaming.

The Synchronize button in the SFTP browser is missing

The Synchronize button is only available in the dual-pane SFTP browser (tab opened via SFTP session or right-click → "Open file transfer here"), not in the SFTP sidebar panel. Make sure both local and remote panels are navigated to the desired folder before synchronizing.

Session restore on startup doesn't work

Check: (1) the option is enabled in Settings → General; (2) the session is still present in connections.json with the same host, protocol, port and user as when the session was open at the time of closing. Sessions opened via Quick Connect (no saved profile) are not restored.

Icons don't appear — only colored dots visible

PNG icons must be present in the icons/ folder next to PCM.py. Check the .png files. If you only have .svg, run bash convert_icons.sh from the PCM folder.

Snippet doesn't appear in the terminal context menu

The context menu with "Insert snippet…" appears only if at least one snippet is configured in the Library. Go to Tools → Snippet Library, add a snippet and try again.

SFTP browser does not connect

The SFTP browser uses paramiko. Check installation:
.venv/bin/python3 -c "import paramiko"
If missing: .venv/bin/pip install paramiko from the gtk3 folder.

Macros are slow / arrive with delay

PCM sends macros via the GDK clipboard (native GTK3) instantly, regardless of command length. If you experience clipboard issues on X11 sessions, check that xclip is installed: which xclip. On Wayland the clipboard is managed natively without external dependencies.

"Export commands.sh" doesn't find commands

Commands typed directly in the terminal are captured only through the terminal log. Make sure the session profile has "Record output to file" enabled and that the session was reconnected after enabling the log. Commands sent via macros and multi-exec are always tracked regardless of the log.

KeePassXC not found / "not running"

Check: (1) KeePassXC is open and the database is unlocked; (2) Browser Integration is enabled in KeePassXC → Tools → Settings → Browser Integration; (3) the Unix socket exists: ls /run/user/$(id -u)/org.keepassxc.KeePassXC.BrowserServer. Use Tools → KeePassXC for the connection test.

Broadcast doesn't reach all terminals

Broadcast only works on VTE embedded terminals (SSH/Telnet/Mosh/Exec tabs open in PCM). RDP and VNC sessions in external windows are not reachable because they don't use the internal terminal.

Where are settings and sessions saved?

  • connections.json — all sessions (plaintext, editable JSON)
  • pcm_settings.json — global settings, shortcuts, recent sessions
  • audit_log.json — connection audit log (metadata, no passwords)
  • /tmp/pcm_logs/ — terminal logs (if enabled, path configurable)

How do I connect to a server with a non-standard SSH port?

Specify the port in the "Port" field of the session dialog or use the format user@host:2222 directly in the Quick Connect bar.

Can I use PCM on multiple machines with the same sessions?

Yes — the connections.json file is portable. Sync it with Dropbox, Nextcloud, Git or copy it manually. Use File → Export sessions for backups and transfers.

⌨ Command line launch

PCM supports opening connections directly from the command line via URI. If PCM is already running, the connection is added as a new tab in the existing window — without asking for the master password again if credentials are already unlocked.

ℹ️ PCM uses D-Bus to detect if it is already open. A second invocation sends the URI to the existing instance via D-Bus and exits immediately — no duplicate PCM windows.

URI format

python3 PCM.py  proto://[user@]host[:port][?options]
ProtocolURI schemeDefault port
SSHssh://22
Telnettelnet://23
RDPrdp://3389
VNCvnc://5900
SFTPsftp://22
FTPftp://21
FTPSftps://21
Moshmosh://60001

Query string options

Only for ad-hoc connections (host not found among saved sessions):

ParameterValuesDescription
modeinternal / externalForce embedded tab in PCM or external window/client
terminalxterm, kitty, …Specific external terminal to use (SSH/Telnet with mode=external only)

Saved session lookup

When a URI is passed, PCM searches for a matching session in this order:

  1. Session name equals the host in the URI (case-insensitive)
  2. Session hostname equals the URI host, same protocol
  3. Session hostname equals the URI host, any protocol
  4. No match → ad-hoc connection using URI parameters

When a saved session is found, all its settings are used (password, key, macros, etc.). User and port from the URI can override the saved values.

Examples — Saved sessions

# Open saved session named "jiraapp" (or with hostname "jiraapp")
python3 PCM.py ssh://jiraapp

# Open saved RDP session "windowsserver"
python3 PCM.py rdp://windowsserver

# Open saved SFTP session "nas"
python3 PCM.py sftp://nas

# Open Telnet session — if PCM is already open, adds a new tab
python3 PCM.py telnet://router-core

# Override the user of saved session "webserver"
python3 PCM.py ssh://root@webserver

Examples — Ad-hoc connections (not saved)

# SSH with user and host
python3 PCM.py ssh://admin@192.168.1.10

# SSH with non-standard port
python3 PCM.py ssh://deploy@prodserver.example.com:2222

# SSH in internal terminal (default)
python3 PCM.py ssh://user@host

# SSH in external terminal
python3 PCM.py 'ssh://user@host?mode=external'

# SSH in specific external terminal
python3 PCM.py 'ssh://user@host?mode=external&terminal=kitty'

# SSH in specific external terminal (another example)
python3 PCM.py 'ssh://user@host?mode=external&terminal=xterm'

# RDP ad-hoc: opens xfreerdp (external by default)
python3 PCM.py rdp://192.168.1.50

# RDP with user, embedded in PCM panel
python3 PCM.py 'rdp://administrator@192.168.1.50?mode=internal'

# VNC external viewer
python3 PCM.py vnc://raspberrypi

# VNC embedded viewer with port
python3 PCM.py 'vnc://192.168.1.60:5901?mode=internal'

# SFTP ad-hoc with user
python3 PCM.py sftp://backup@nas.lan

# FTP ad-hoc
python3 PCM.py ftp://ftp.example.com

# FTPS ad-hoc
python3 PCM.py ftps://secure-ftp.example.com

# Telnet with custom port
python3 PCM.py telnet://cisco-switch:2323

# Mosh ad-hoc
python3 PCM.py mosh://user@remotehost

Useful aliases in ~/.bashrc

# Quick access to frequent hosts
alias jira='python3 ~/PCM/gtk3/PCM.py ssh://jiraapp'
alias nas='python3 ~/PCM/gtk3/PCM.py sftp://nas'
alias vpn-gw='python3 ~/PCM/gtk3/PCM.py ssh://vpn-gateway'

# RDP to Windows workstation
alias winpc='python3 ~/PCM/gtk3/PCM.py rdp://workpc'

# Generic function: open any SSH host in PCM
pcm() { python3 ~/PCM/gtk3/PCM.py "ssh://$1"; }

Behavior with encrypted credentials

PCM is not open

PCM starts normally, shows the master password unlock dialog, and after unlocking opens the requested connection automatically. No need to re-run the command.

PCM is already open and unlocked

The connection opens immediately as a new tab. No master password prompt.

💡 Make sure to start PCM at least once with the new code before using the URI feature: the first launch registers the D-Bus flag needed for URI forwarding to the existing instance.
